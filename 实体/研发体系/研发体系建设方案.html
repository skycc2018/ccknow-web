二、研发流程规范

2.1 全流程概览

需求提出 → 需求评审 → 技术方案 → 开发实现 → Code Review → 测试 → 发布 → 复盘
   (PM)      (全员)      (架构师)     (FDE)       (架构师+FDE)   (FDE)   (FDE)    (全员)

2.2 各阶段详细规范

Phase 1：需求管理（产品总监 + 四洋）

环节 输入 输出 负责人 时限
需求收集 业务痛点、用户反馈、竞品分析 需求池（Backlog） 四洋 持续
需求分析 需求池 PRD文档 四洋 2-3天
需求评审 PRD 评审纪要、修改清单 产品总监 1天
优先级排序 评审通过的PRD 迭代计划 产品总监 半天

PRD 规范： - 必须包含：背景、目标、用户故事、功能清单、验收标准 - 可选包含：原型图、流程图、数据埋点 - 模板位置：知识库/templates/PRD-模板.md

Phase 2：技术设计（架构师）

环节 输入 输出 负责人 时限
技术方案 PRD 技术方案文档 架构师 1-2天
方案评审 技术方案 评审纪要 架构师+CIO 半天
任务拆分 技术方案 开发任务清单 FDE 半天

技术方案文档必须包含： 1. 架构设计（系统图、数据流图） 2. 接口设计（API 定义、数据模型） 3. 技术选型（选型对比表、决策依据） 4. 风险评估（技术风险、应对策略） 5. 性能预估（QPS、存储、带宽）

Phase 3：开发实现（FDE）

环节 输入 输出 负责人 时限
代码开发 开发任务 代码 + 单元测试 FDE 按迭代计划
Code Review PR Review 意见 架构师 1天内
文档更新 代码变更 技术文档更新 FDE 随代码提交

代码提交规范：

<type>(<scope>): <subject>

<body>

<footer>
Type 含义
feat 新功能
fix Bug 修复
docs 文档更新
style 代码格式（不影响功能）
refactor 重构
test 测试相关
chore 构建/工具相关

Phase 4：测试与发布（FDE + 架构师）

环节 输入 输出 负责人 时限
自测 代码 自测报告 FDE 开发完成后
集成测试 合并后的代码 测试报告 FDE 1天
发布准备 测试通过 发布清单 FDE 半天
灰度发布 发布清单 线上验证 FDE 按策略
全量发布 灰度通过 上线完成 FDE 半天

发布流程： 1. 功能开关（Feature Flag）控制 2. 灰度发布：5% → 20% → 50% → 100% 3. 回滚策略：自动监控，异常自动回滚 4. 发布窗口：工作日 10:00-12:00 或 14:00-16:00

Phase 5：复盘迭代（全员）

环节 输入 输出 负责人 时限
迭代复盘 迭代数据 复盘纪要 我（助理） 迭代结束后1天
经验沉淀 复盘纪要 知识库更新 全员 持续

三、项目管理机制

3.1 迭代节奏

项目类型 迭代周期 站会频率 评审频率
核心产品（ShareLLM） 2周 每日 每迭代
支撑项目（内部工具） 1周 按需 每迭代
探索项目（AI实验） 3周 每周2次 每迭代

3.2 会议机制

会议 频率 参与人 时长 输出
晨会（Stand-up） 每日 全员 15min 阻塞项清单
需求评审会 每迭代启动 PM+架构师+FDE 1h 迭代需求清单
技术方案评审 按需 架构师+FDE 1-2h 技术方案文档
迭代复盘会 迭代结束 全员 1h 复盘纪要
月度规划会 每月 全员+CIO 2h 月度计划

3.3 优先级定义

优先级 定义 响应时间
P0 阻塞主流程，必须立即处理 2小时内
P1 影响核心功能，本周必须完成 1天内
P2 影响体验，本迭代完成 按迭代计划
P3 优化项，有空再做 排入后续迭代

四、技术规范

4.1 技术栈标准

当前项目（ShareLLM / 菜洋洋商城）统一技术栈：

层级 选型 说明
前端 React 19 + TypeScript + Ant Design 统一UI框架
后端 Go 1.22 (Gin/GORM) 高性能网关
AI服务 Python (FastAPI) LLM生态更成熟
数据库 PostgreSQL + Redis 主从分离
向量库 Milvus / PGVector RAG能力
消息队列 Redis Stream / NATS 异步处理
容器化 Docker + Docker Compose 开发/测试
部署 K8s (生产) 云原生
监控 Prometheus + Grafana 指标监控
日志 ELK / Loki 日志收集

4.2 代码规范

必读规范： - Go: 遵循 gofmt + golint - TypeScript: ESLint + Prettier - Python: PEP8 + Black - 代码注释率：关键逻辑必须注释 - 单测覆盖率：核心逻辑 ≥ 70%

分支模型：

main (生产)
  ↑
release/v1.x (预发)
  ↑
develop (开发)
  ↑
feature/xxx (功能)
  ↑
hotfix/xxx (紧急修复)
