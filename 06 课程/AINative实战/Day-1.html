🎓 AI Native 实战课程 - Day 1: 第一章 - AI Enhanced vs AI Native

📖 今日导读

2023 年 ChatGPT 点燃生成式 AI 以来，企业最容易踩的第一个坑就是：把 AI 当成一个功能按钮。在原有产品里加一个「AI 写作助手」、「AI 总结」，这叫 AI Enhanced（AI 增强）。而 AI Native（AI 原生）是从设计之初就让 AI 成为核心引擎，重新定义交互范式：从「人适应机器」变成「机器理解人」。

今天这一章是整门课程的地基。只有先分清两者的边界，你才能在后续设计 Prompt 即接口、上下文即状态、工具调用与 Agent 时做出正确的产品决策。我们会结合 2026 年最新的产品动态，用一个可运行的 CRM 案例，让你直观感受两种范式的差距。

🔍 核心概念

1. AI Native 的三个特征

在 AI Native 深度解析：从概念到落地 中，AI Native 被定义为「从设计之初就以 AI 为核心驱动力的产品」。它必须同时具备三个特征：

特征 传统 +AI AI Native
意图驱动 点菜单、填表单、筛选器 说自然语言，系统理解目标
持续学习 功能一成不变 每次交互都优化下次交互
自主行动 等待指令才执行 主动规划并调用工具完成目标

三者缺一不可。只有当 AI 从「会说话」进化到「会办事」，才能称之为真正的 AI Native 。

2. AI Enhanced vs AI Native vs AI First

《AI Native：从概念到落地》- 第一章 把 AI 产品分为三层：

  • AI Enhanced（AI 增强）：给现有产品加个 AI 按钮。例如 Word 里的「AI 写作助手」。交互方式没变，人仍然在适应机器。
  • AI Native（AI 原生）：围绕 AI 重新设计产品。例如 Notion 从「文档工具」变成「AI 工作空间」，人只需表达意图。
  • AI First（AI 优先）：AI 成为核心，人做审核。例如 Devin 自主完成编程任务。

2024-2026 年的机会在第二层。 第一层是红海，第三层还不成熟（技术、信任、监管都未就绪），AI Native 是蓝海。

3. 2026 年的关键趋势

结合 Anthropic 2026 年5 月发布的 The Founder’s Playbook 和 Block 6 月实践，行业已经从「用 AI 写代码」进入「AI-native engineering / AI-native org 」阶段：

  • Agentic workflow 成为主流：AI 不再是「回答问题」，而是「规划任务 → 调用工具 → 执行 → 反馈」的循环。
  • MCP（Model Context Protocol）成为标配：Anthropic 与 Block 联合推动的 MCP 正在成为连接 AI Agent 与企业数据源的行业标准。
  • 组织范式变革：AI 吃掉的不是某个岗位，而是「翻译层」（需求 → 报告 → JIRA → PR → 发布说明的无限转换）。未来组织的核心竞争力是「如何让人类从执行者变成 orchestrator」。

💼 最新案例

案例一：Block Builderbot —— AI-native 的工程协同

2026 年6 月17 日，Block 发布了 Builderbot：一套 AI-native 工具集，目标是改变大规模组织的开发方式。

  • 数据：Block 100% 的工程师定期使用 AI；Builderbot 每天执行超过 20 万次 操作，每周合并约 1,500 个 Pull Request，约占 Block 全部生产代码变更的 15%。
  • 交互范式：工程师在 Slack 里 @builderbot，用自然语言描述 bug 修复、跨服务迁移或新功能，AI 在线程里完成调研、规划、写代码、提交 PR、监控 CI 和迭代。
  • 关键差异：它不是「一个仓库里的代码补全」，而是能跨越 Block 数亿行代码、数百个服务和整套工程规范的编排层。
  • 开源：Builderbot 建立在 Block 开源的 goose Agent 框架上，并使用 MCP 作为 AI 与工具/数据源的连接协议。

这正是 AI Native 的典型特征：不是「给现有工作流程加 AI 」，而是「让 AI 成为工作流程本身」。

案例二：Perplexity —— AI-native 搜索的持续进化

Perplexity 是 AI Native 搜索的标杆。与传统搜索「给链接」不同，Perplexity 「给答案」，并为每个数据标注来源。

  • 2026 年新动态：
  • 5 月推出 Mac 应用，并上线 Personal Computer —— 一个混合本地与云端的 AI Agent，被定位为「个人编排器」。
  • 6 月 Perplexity CEO 在 CNBC 采访中确认计划 2028 年 IPO。
  • 其系统会根据任务自动选择最优模型，并考虑成本。
  • 关键差异：传统搜索你输入关键词，然后自己筛选阅读；Perplexity 让你直接提问，它理解问题、检索多个来源、生成带引用的答案，并支持上下文追问。

Perplexity 的竞争力不是「更快的 Google」，而是 「从找信息到获取知识」的范式转移。

案例三（开源项目）：Vessel Browser

Vessel 是 2026 年出现的开源 AI-native 浏览器，它的定位不是「带 AI 功能的浏览器」，而是「Agent 的浏览器运行时」：

  • Agent 是主要操作者，人类通过可见界面监督、审计和干预。
  • 支持持久状态、MCP 控制、书签、检查点、动作撤销，适合长时间运行的 Agent 工作流。
  • GitHub: unmodeled-tyler/vessel-browser

💻 实战代码

下面这个 Python 脚本模拟了一个简单的 CRM 场景，对比 AI Enhanced（固定筛选）与 AI Native（意图驱动）的差异。

  • 不需要外部 API 即可运行本地演示版本。
  • 如果设置了 OPENAI_API_KEY，脚本会自动切换到 OpenAI 的 Function Calling 模式，体验真正的 AI Native 交互。
import os
import re
import json
import sqlite3
from datetime import datetime, timedelta

DB_PATH = "crm_demo.db"

# ===== 1. 数据准备 =====
def init_db():
    """初始化示例数据库"""
    conn = sqlite3.connect(DB_PATH)
    c = conn.cursor()
    c.execute("DROP TABLE IF EXISTS customers")
    c.execute('''
        CREATE TABLE customers (
            id INTEGER PRIMARY KEY,
            name TEXT,
            company TEXT,
            last_visit TEXT,
            status TEXT,
            deal_size INTEGER
        )
    ''')
    today = datetime.now().date()
    sample = [
        (1, "张三", "智云科技", (today - timedelta(days=5)).isoformat(), "pending", 80000),
        (2, "李四", "碧海集团", (today - timedelta(days=10)).isoformat(), "pending", 120000),
        (3, "王五", "星火网络", (today - timedelta(days=20)).isoformat(), "won", 50000),
        (4, "赵六", "金沙云", (today - timedelta(days=3)).isoformat(), "pending", 30000),
        (5, "钱七", "青山科技", (today - timedelta(days=12)).isoformat(), "pending", 60000),
    ]
    c.executemany("INSERT INTO customers VALUES (?,?,?,?,?,?)", sample)
    conn.commit()
    return conn

# ===== 2. AI Enhanced: 固定筛选器 =====
def ai_enhanced_find_customers(status=None, visited_after=None, min_deal=None):
    """传统 +AI 方式：用户必须知道要点哪些筛选条件"""
    conn = sqlite3.connect(DB_PATH)
    c = conn.cursor()
    query = "SELECT * FROM customers WHERE 1=1"
    params = []
    if status:
        query += " AND status = ?"
        params.append(status)
    if visited_after:
        query += " AND last_visit >= ?"
        params.append(visited_after)
    if min_deal:
        query += " AND deal_size >= ?"
        params.append(min_deal)
    c.execute(query, params)
    rows = c.fetchall()
    conn.close()
    return rows

# ===== 3. AI Native: 意图驱动 =====
def _local_parse_intent(text: str) -> dict:
    """本地意图解析器（无需 API）"""
    cond = {}
    text = text.lower()

    # 状态
    if re.search(r"未跟进|待跟进|没跟进|pending", text):
        cond["status"] = "pending"
    elif re.search(r"已成交|won", text):
        cond["status"] = "won"

    # 拜访时间：上周 / 过去 N 周
    if re.search(r"上周|last week", text):
        cond["visited_after"] = (datetime.now() - timedelta(weeks=1)).date().isoformat()
    elif m := re.search(r"过去\s*(\d+)\s*周|last\s+(\d+)\s+weeks?", text):
        weeks = int(m.group(1) or m.group(2))
        cond["visited_after"] = (datetime.now() - timedelta(weeks=weeks)).date().isoformat()

    # 合同金额：大于 X 万 / 大于 X
    if m := re.search(r"大于\s*(\d+)\s*(万?)", text):
        value = int(m.group(1))
        if m.group(2) == "万":
            value *= 10000
        cond["min_deal"] = value

    return cond

def _ai_native_with_openai(text: str) -> tuple:
    """使用 OpenAI Function Calling 解析自然语言"""
    try:
        from openai import OpenAI
    except ImportError:
        raise RuntimeError("请先安装 openai: pip install openai")

    client = OpenAI()
    tools = [{
        "type": "function",
        "function": {
            "name": "query_customers",
            "description": "根据用户自然语言查询 CRM 客户",
            "parameters": {
                "type": "object",
                "properties": {
                    "status": {"type": "string", "enum": ["pending", "won"], "description": "客户跟进状态"},
                    "visited_after": {"type": "string", "description": "最早拜访日期，ISO-8601 格式"},
                    "min_deal": {"type": "integer", "description": "合同金额下限"},
                },
                "required": [],
            },
        },
    }]

    response = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[{"role": "user", "content": text}],
        tools=tools,
        tool_choice="auto",
    )
    message = response.choices[0].message
    if message.tool_calls:
        args = json.loads(message.tool_calls[0].function.arguments)
        return ai_enhanced_find_customers(**args), args
    return [], {}

def ai_native_find_customers(text: str) -> tuple:
    """根据环境自动选择解析方式"""
    if os.getenv("OPENAI_API_KEY"):
        print("🔄 检测到 OPENAI_API_KEY，使用 LLM Function Calling 解析意图...")
        return _ai_native_with_openai(text)
    print("🔄 未检测到 OPENAI_API_KEY，使用本地意图解析器...")
    cond = _local_parse_intent(text)
    return ai_enhanced_find_customers(**cond), cond

# ===== 4. 演示 =====
if __name__ == "__main__":
    init_db()
    query = "帮我列出上周拜访过但还没跟进的客户，合同金额大于5万"

    print("─" * 60)
    print("🔴 AI Enhanced 方式：用户需要自己翻译成筛选条件")
    print("─" * 60)
    enhanced_rows = ai_enhanced_find_customers(
        status="pending",
        visited_after=(datetime.now() - timedelta(weeks=1)).date().isoformat(),
        min_deal=50000,
    )
    for r in enhanced_rows:
        print(r)

    print("\n")
    print("─" * 60)
    print("🟢 AI Native 方式：用户只需说人话")
    print("─" * 60)
    native_rows, conditions = ai_native_find_customers(query)
    print(f"解析出的条件: {json.dumps(conditions, ensure_ascii=False)}")
    for r in native_rows:
        print(r)

关键对比：AI Enhanced 要求人学会系统的语言（状态、日期、金额）；AI Native 让系统学会人的语言。

✅ 最佳实践

结合 Anthropic 《The Founder’s Playbook》、Block Builderbot 实践以及 AI-native org 的组织研究，总结出 6 条可操作的指南：

  1. 从用户任务出发，而非功能 先写下用户想完成的 Job-to-be-Done，再决定 AI 放在哪个环节。不要问「我能加什么 AI 功能」，而要问「如果机器理解人，这个任务可以怎么被消灭」。

  2. 把自然语言当作新的 API 在 AI Native 产品中，Prompt 和 Function Calling 就是接口。为 AI 定义清晰的工具边界（参数、返回值、异常处理），比写一个完美的对话更重要。

  3. 设计人机协作的连续谱 从 AI Enhanced → AI Native → AI First 是一个渐进过程。无论哪一层，都要有：

  4. 置信度评分
  5. 人工确认/审批节点

  6. 随时人工接管机制

  7. 以数据和上下文为护城河 模型能力会趋同，但企业私有数据、知识库、业务流程不会。通过 RAG、长期记忆、MCP 连接企业系统，才能构建可持续的差异化。

  8. 在组织里压缩「翻译层」 AI 最先取代的不是某个岗位，而是「需求 → PRD → 工单 → PR → 发布说明 → 汇报」的转换工作。让人类从执行者变成 orchestrator，这也是 Block Builderbot 15% PR 由 Agent 合并背后的组织启示。

  9. 安全与信任先行 Block 明确约定 Builderbot 只操作源代码和系统配置，不接触用户数据、支付信息和 PII。这是 AI Native 落地时必须建立的信任基础。

🤔 思考题

  1. 你最近使用的一款产品属于 AI Enhanced / AI Native / AI First 中的哪一层？如果将它升级到 AI Native 层，用户会用什么自然语言指令来代替点击按钮？

  2. 你的工作中哪个环节是「人适应机器」的？请描述如何用 意图驱动 + 工具调用 重构它，并列出需要调用的至少 2 个外部工具。

  3. 假设你的团队即将上线一个每天自动合并 15% 代码变更的 AI Native 助手。为了确保质量与安全，你会设计哪些 置信度评分、人工审批和撤回机制？
