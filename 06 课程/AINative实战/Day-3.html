🎓 AI Native 实战课程 - Day 3: 第三章 - Prompt 即接口

📖 今日导读

在 [[AI Native 深度报告]] 中，有一条被重复强调的设计原则：Prompt 即接口。它的意思是：在 AI Native 产品里，自然语言不再只是「输入方式」，而是系统之间的「调用契约」。

传统软件中，你要查询一个客户，需要调用 REST API：GET /customers?id=123。而在 AI Native 产品中，你可能只需说一句话：「帮我找出上周拜访过但还没跟进的客户」，这句话就是一次对 LLM 的接口调用。

今天这一章，我们不仅要学会「写好 Prompt」，更要学会「设计 Prompt 接口」：如何让自然语言可靠地驱动后端系统，如何让输出可验证、可测试、可维护。这是从「会用 AI 的产品经理」进化到「会设计 AI Native 产品」的关键一步。

🔍 核心概念

1. 自然语言成为新的 API

[[《AI Native：从概念到落地》- 书籍规划]] 在第三章开篇就指出：AI Native 产品的第一个原理就是「自然语言成为新的 API」。

这里有三层含义：

维度 传统 API Prompt 即接口
调用方式 HTTP + 固定参数 自然语言 + 上下文
输入灵活度 严格的字段和类型 不固定格式，系统理解意图
输出形式 JSON / XML 等确定格式 可以是文本、JSON、代码、决策、行动
错误处理 400 / 500 状态码 模糊回答、幻觉、偏离任务

这不是说 REST API 消失了，而是说：LLM 在用户与底层系统之间，新增了一层「自然语言调度层」。用户说人话，LLM 理解后转换成底层 API 调用、数据库查询、文档生成或多步骤任务规划。

2. Prompt 不是「咒语」，是「接口契约」

很多人刚学 Prompt Engineering 时，会把它当成「骗模型说出更好答案的技巧」。但在 AI Native 产品里，Prompt 是一份需要设计、维护、测试和版本控制的「接口契约」。

一份好的 Prompt 接口应该包含：

  1. 谁在用（Role）：系统的身份和能力边界
  2. 在什么场景下用（Context）：业务背景、用户、约束条件
  3. 要完成什么任务（Intent）：明确的目标描述
  4. 输出必须满足什么结构（Output Schema）：JSON Schema、Pydantic 模型、函数参数
  5. 失败时怎么办（Fallback）：缺少信息、置信度低、幻觉时的处理策略

把 Prompt 当接口看待，你就不会只问「这个 Prompt 能不能让模型回答得更好」，而会问「这个 Prompt 能不能在生产环境里稳定地被调用 10 万次」。

3. 结构化 Prompt：CRISPE 框架

[[《AI Native：从概念到落地》- 书籍规划]] 在第三章提出使用 CRISPE 框架设计结构化 Prompt。它是一种把「零散的提示词」转变为「可维护的接口文档」的方法。

字母 英文 中文 作用
C Context 上下文 说明任务背景、系统约束、用户场景
R Role 角色 定义 AI 应该扮演什么身份
I Intent 意图 明确这次调用的目标
S Specifics 具体要求 输出结构、长度、格式、约束
P Examples 示例 少样本学习，让模型理解什么是好的输出
E Extras 额外 安全限制、输出前置条件、特殊说明

CRISPE 的价值不在于「每次都要写全部六部分」，而在于提供一套「接口设计检查清单」：当你写完一个 Prompt 时，可以逐条检查是否说清楚了「谁、在哪、干什么、怎么做、输出什么」。

4. 从「会写 Prompt」到「会设计 Prompt 接口」

Prompt Engineering 的成熟度可以分为四层：

第一层：随意问答
   “你是一个有用的助手，帮我写东西」

第二层：角色 + 任务
   “你是一名产品经理，请为以下功能写用户故事」

第三层：结构化 + 少样本
   系统提示词 + 业务上下文 + 输出格式约束 + 示例

第四层：接口化 + 可测试 + 可监控
   Prompt 版本管理 + 评估集 + A/B 测试 + 成本/延迟监控

2025-2026 年，行业的进化正在从第二、三层迈向第四层。单点的 Prompt 调优已经不够，团队需要：

  • Prompt Registry（注册中心）：集中管理所有生产级 Prompt
  • Eval Suite（评估套件）：每次修改 Prompt 前都要跑一遍测试集
  • A/B 测试：对比不同 Prompt 版本的表现
  • 成本监控：跟踪 Token 消耗、缓存命中率、模型级联策略

5. 输出结构化：让 Prompt 真正可被程序调用

Prompt 要想从「对话」变成「接口」，必须解决一个核心问题：让 LLM 输出可解析的结构。2024-2025 年，所有主流模型和平台都在加强这一能力：

  • OpenAI Structured Output / JSON Mode / Function Calling：强制模型输出符合 JSON Schema
  • Anthropic Tool Use：让 Claude 根据描述选择并调用工具
  • Pydantic / Instructor：用 Python 类型系统绑定 LLM 输出，让调用方像调用普通函数一样简单

💡 关键认知：AI Native 产品的前端不是 Chat UI，而是「自然语言调度层」；后端不是一个巨大的 LLM，而是「被 Prompt 驱动的一组工具和数据源」。

💼 最新案例

案例一：OpenAI Operator 与 Deep Research —— 自然语言成为网页操作接口

2025 年1 月，OpenAI 发布了 Operator（研究预览版）。它不是一个新的聊天窗口，而是一个能够用自然语言驱动浏览器完成任务的 Agent。

  • 产品定位：用户只需说出目标，例如「给我在 DoorDash 上订一份披萨和水」，Operator 就会自主打开网页、选择餐厅、加入购物车、填写地址、完成支付。
  • 技术特点：Operator 基于一个名为 CUA（Computer-Using Agent）的模型，能够看到屏幕、理解界面元素并执行鼠标键盘操作。
  • Prompt 即接口启示："Prompt" 不再是给 LLM 的一段文字，而是给整个操作系统的高层指令。这与传统 RPA 需要人工录制脚本形成鲜明对比。

同时期的 OpenAI Deep Research则展示了另一种形态：用户输入一个研究主题的 Prompt，系统自动浏览数十个网页并生成综合报告。在 GAIA 基准测试中，Deep Research 达到了 72.57%。

这些产品的共同点是：自然语言取代了点击、表单和脚本，成为调度外部世界的主要接口。

案例二：Cursor Agent、Claude Code 与 OpenAI Codex CLI —— 自然语言成为开发环境接口

2025 年是 AI 编程的「Agent 化」之年。三个产品代表了同一个趋势：

  • Cursor Agent / Composer（2025 年全面开放）：Anysphere 的 Cursor 从「代码补全」进化为「代码编辑器里的 Agent」。开发者可以用自然语言描述需求，Cursor 会自动修改多个文件、运行终端命令、自我修复错误。
  • Claude Code（2025 年2 月发布）：Anthropic 推出的终端编码 Agent，能在整个代码库中探索、编辑、测试、提交代码。它的核心交互不是键盘快捷键，而是自然语言对话。

  • OpenAI Codex CLI（2025 年5 月发布）：OpenAI 将其最新的 Codex 模型放进终端，开发者直接用自然语言命令它写代码、运行测试、审查 PR。在 SWE-bench Verified 上达到 72.7%。

  • 数据与趋势：根据 GitHub 和各家公司的分享，2026 年领先团队已经有 15%-30% 的代码变更来自 AI Agent。

  • Prompt 即接口启示：开发者不再需要记住某个 IDE 功能在哪个菜单里，只需用自然语言描述任务。IDE 的界面没有消失，但主要的操作接口已经变成自然语言。

案例三：Manus —— 自然语言成为通用任务接口

2025 年3 月，Monica.im 发布了 Manus，定位为通用型 AI Agent。用户只需输入一个任务描述，Manus 就会在云端虚拟环境中自主规划、调用工具、执行并返回结果。

  • 关键数据：Manus 官方声称在 GAIA benchmark上取得了 SOTA，超越 OpenAI Deep Research。
  • 产品特点：不是「问答」，而是「办事」。它能自动分解任务、搜索信息、撰写报告、甚至写代码和生成可视化结果。
  • Prompt 即接口启示：当任务越复杂，「怎么写 Prompt」就越像「怎么设计一个高层任务 DSL」。结构化、步骤分解、工具调用规范，才能让自然语言真正可靠地驱动复杂任务。

三个案例的共同结论：2025 年以来，最具震撼力的 AI 产品，都是把自然语言从「聊天输入」升级为「操作系统的主要接口」。

💻 实战代码

下面的 Python 脚本演示了如何把自然语言封装成一个可调用的接口。我们会：

  1. 用 Pydantic 定义输出结构（接口契约）
  2. 用 CRISPE 框架 组织系统 Prompt
  3. 使用 OpenAI 的 Structured Output 能力（需要 OPENAI_API_KEY）
  4. 无 API Key 时自动回退到本地规则解析，保证可立即运行
"""
Day 3 实战代码：Prompt 即接口
目标：把"自然语言指令"封装成可调用的接口，输出结构化数据。

运行方式：
  1. 不设置 OPENAI_API_KEY：使用本地规则解析，可立即看到效果
  2. 设置 OPENAI_API_KEY：调用 GPT-4o-mini 的 Structured Output

依赖：pip install pydantic openai
"""

import os
import json
from typing import Literal
from pydantic import BaseModel, Field


# ==================== 1. 定义接口契约（输出结构） ====================
class SupportTicket(BaseModel):
    """客服工单结构化输出

    这就是 Prompt 即接口的"返回类型"。
    调用方只需关心这个 Pydantic 模型，而不用关心模型到底输出了什么文本。
    """
    intent: str = Field(..., description="用户意图，不超过10个字")
    urgency: Literal["low", "medium", "high"] = Field(..., description="紧急程度")
    product: Literal["云服务器", "对象存储", "CDN", "数据库", "未知"] = Field(
        ..., description="相关产品"
    )
    action: str = Field(..., description="建议采取的行动")
    reply: str = Field(..., description="给用户的回复草稿")


# ==================== 2. 用 CRISPE 框架组织系统 Prompt ====================
CRISPE_SYSTEM_PROMPT = """\
# Context（上下文）
你是一家云服务商的一线客服助手。服务产品包括：云服务器、对象存储、CDN、数据库。
公司 SLA 规定：涉及服务中断、数据丢失、安全漏洞的反馈必须在 30 分钟内人工介入。

# Role（角色）
你是经验丰富的一线客服工程师，擅长从用户描述中快速提炼关键信息并生成工单。

# Intent（意图）
分析用户的反馈，输出结构化的客服工单。

# Specifics（具体要求）
- intent：简短概括用户意图，不超过 10 个字
- urgency：根据问题影响范围选择 low / medium / high
- product：从 {云服务器, 对象存储, CDN, 数据库, 未知} 中选择最相关的一个
- action：给出具体、可执行的下一步行动
- reply：用中文生成一段礼貌、专业的用户回复草稿

# Examples（示例）
用户：我的网站打不开了，已经持续半小时。
输出：
{
  "intent": "服务中断",
  "urgency": "high",
  "product": "云服务器",
  "action": "立即检查实例运行状态并联系值班工程师",
  "reply": "非常抱歉给您带来不便，我们已经将您的问题标记为高优先级，工程师会在 30 分钟内与您联系。"
}

用户：请问 CDN 怎么配置缓存规则？
输出：
{
  "intent": "配置咨询",
  "urgency": "low",
  "product": "CDN",
  "action": "发送 CDN 缓存配置文档链接",
  "reply": "您好，CDN 缓存配置可以参考我们的官方文档，相关链接已发送至您的邮箱。"
}

# Extras（额外约束）
只输出合法的 JSON，不要添加 markdown 代码块或任何解释。
"""


# ==================== 3. LLM 实现（需要 OPENAI_API_KEY） ====================
def parse_ticket_with_llm(user_input: str) -> SupportTicket:
    """使用 OpenAI Structured Output 解析工单。

    这里就是把 Prompt 当作接口调用的关键：
    输入是自然语言，输出是类型安全的 Pydantic 对象。
    """
    from openai import OpenAI

    client = OpenAI()
    response = client.beta.chat.completions.parse(
        model="gpt-4o-mini",
        messages=[
            {"role": "system", "content": CRISPE_SYSTEM_PROMPT},
            {"role": "user", "content": user_input},
        ],
        response_format=SupportTicket,
    )
    return response.choices[0].message.parsed


# ==================== 4. 本地规则实现（无需 API Key） ====================
def parse_ticket_local(user_input: str) -> SupportTicket:
    """无 API Key 时的本地规则解析，用于快速演示接口形态。"""
    text = user_input.lower()
    if any(k in text for k in ["打不开", "无法访问", "中断", "down", "宕机"]):
        return SupportTicket(
            intent="服务中断",
            urgency="high",
            product="云服务器",
            action="立即检查实例运行状态并联系值班工程师",
            reply="非常抱歉给您带来不便，我们已经将您的问题标记为高优先级，工程师会在 30 分钟内与您联系。",
        )
    if any(k in text for k in ["cdn", "缓存"]):
        return SupportTicket(
            intent="配置咨询",
            urgency="low",
            product="CDN",
            action="发送 CDN 缓存配置文档链接",
            reply="您好，CDN 缓存配置可以参考我们的官方文档，相关链接已发送至您的邮箱。",
        )
    if any(k in text for k in ["数据库", "sql", "丢数据"]):
        return SupportTicket(
            intent="数据库问题",
            urgency="medium",
            product="数据库",
            action="检查数据库连接和慢查询日志",
            reply="您好，已记录您的数据库问题，技术团队会尽快排查。",
        )
    return SupportTicket(
        intent="一般咨询",
        urgency="low",
        product="未知",
        action="记录工单并转交人工客服",
        reply="您好，已收到您的反馈，我们会尽快处理。",
    )


# ==================== 5. 统一的 Prompt 接口入口 ====================
class PromptInterface:
    """把自然语言封装成可调用的接口。

    调用方不需要知道底层是 LLM 还是规则，
    只需传入自然语言，就能获得结构化的 SupportTicket。
    """

    def __init__(self):
        self.use_llm = bool(os.getenv("OPENAI_API_KEY"))

    def parse(self, user_input: str) -> SupportTicket:
        if self.use_llm:
            print("  🔄 检测到 OPENAI_API_KEY，使用 LLM Structured Output 解析...")
            return parse_ticket_with_llm(user_input)
        print("  🔄 未检测到 OPENAI_API_KEY，使用本地规则解析...")
        return parse_ticket_local(user_input)


# ==================== 6. 运行演示 ====================
if __name__ == "__main__":
    interface = PromptInterface()

    test_inputs = [
        "我的网站突然打不开了，已经半个小时，客户都在投诉！",
        "请问 CDN 的缓存规则怎么配置？",
        "数据库连接时快时慢，怀疑是慢查询导致的。",
    ]

    for text in test_inputs:
        print("\n" + "=" * 60)
        print(f"📝 用户输入：{text}")
        ticket = interface.parse(text)
        print(json.dumps(ticket.model_dump(), ensure_ascii=False, indent=2))

代码启示：PromptInterface 封裀了底层实现，对调用方暴露出一个简单的接口：自然语言 -> SupportTicket。这正是 AI Native 产品的典型设计——调用方不需要知道 LLM 存在，只需使用自然语言跟系统交互。

✅ 最佳实践

结合 2025-2026 年 Prompt Engineering 的最新进展与行业最佳实践，总结 6 条可操作的指南：

  1. 把 Prompt 当作代码一样管理 不要把 Prompt 写死在业务代码里，而是放入单独的文件或 Prompt Registry，支持版本控制、A/B 测试和回滚。工具可选：Langfuse（29.7k ⭐）、PromptLayer、LangSmith、Phoenix。

  2. 优先使用结构化输出 无论是 JSON Mode、Function Calling，还是 Pydantic 绑定的 Structured Output，都应该成为默认选择。这能大幅减少解析失败和后端处理成本。推荐使用 Instructor（13.2k ⭐）或 PydanticAI 进行类型安全的 LLM 调用。

  3. 用 CRISPE 或类似框架设计每一个生产级 Prompt 不要靠灵感写 Prompt。每个 Prompt 都应该经过：Context → Role → Intent → Specifics → Examples → Extras 的检查，确保它是一份可传承、可审查的接口文档。

  4. 建立 Eval Suite，每次修改 Prompt 前都要跑测试 Prompt 的小改动可能导致产生意想不到的回归。使用 promptfoo（22.6k ⭐）等工具建立测试集，自动检查输出的格式、事实性、安全性。

  5. 对长上下文和重复调用使用缓存 OpenAI 和 Anthropic 都已支持 Prompt Caching，对于长系统提示词和多轮对话可以明显降低成本。同时，合理设计系统提示词，避免每次都传递大段重复的上下文。

  6. 在复杂场景下使用自动优化框架 当需要从几十个后选示例中选出最优的 few-shot 时，手动调试已经不可行。使用 DSPy（35.4k ⭐）等框架，用模块化程序替代手写 Prompt，让系统自动寻找最优的指令和少样本组合。

🤔 思考题

  1. 回顾你的产品或工作流程：如果把某个功能的接口从「表单/按钮」改成「自然语言」，用户会怎么说？这个自然语言提示词需要包含哪些必备信息，才能让 AI 准确地执行任务？

  2. 在你的团队里，目前的 Prompt 是如何管理的？是散落在各个脚本和 Jupyter Notebook 里，还是有统一的版本控制和评估流程？如果要建立一个 Prompt Registry，你会为每个 Prompt 定义哪些元信息？

  3. 结构化输出能大幅提升 AI 的可靠性，但它也有边界（例如模型可能填出「看起来合理但实际错误」的字段）。请设计一个生产环境中的「结构化输出验证流程」，包括：如何验证输出、如何处理异常、如何在失败时保持系统可用。
