代码规范

适用：ShareLLM、菜洋洋商城等所有项目 版本：v1.0

一、Git 规范

1.1 分支模型

main        ← 生产环境，只能由 release 合并
release/*   ← 预发环境，测试通过后合并到 main
develop     ← 开发主分支，功能分支合并目标
feature/*   ← 功能分支，从 develop 检出
hotfix/*    ← 紧急修复，从 main 检出

1.2 提交信息规范

<type>(<scope>): <subject>

<body>

<footer>

Type 列表：

Type 含义 示例
feat 新功能 feat(auth): 添加 JWT 登录
fix Bug 修复 fix(api): 修复用户列表分页错误
docs 文档更新 docs(readme): 更新部署说明
style 代码格式 style(lint): 修复 ESLint 警告
refactor 重构 refactor(service): 提取公共逻辑
test 测试相关 test(unit): 添加用户模块单测
chore 构建/工具 chore(ci): 添加 Docker 构建

Subject 规范： - 不超过 50 个字符 - 使用祈使句（"添加" 而非 "添加了"） - 首字母小写 - 结尾不加句号

1.3 提交频率

  • 每个 commit 只做一件事
  • 每天下班前必须 push
  • PR 前 rebase 整理 commit

二、代码风格

2.1 Go 规范

  • 使用 gofmt 自动格式化
  • 包名全小写，无下划线
  • 接口名以 er 结尾（Reader, Writer）
  • 错误处理：不忽略错误，不滥用 panic
// ✅ Good
func GetUser(ctx context.Context, id int64) (*User, error) {
    user, err := db.Query(ctx, id)
    if err != nil {
        return nil, fmt.Errorf("query user %d: %w", id, err)
    }
    return user, nil
}

// ❌ Bad
func GetUser(id int) *User {
    return db.Query(id) // 忽略错误
}

2.2 TypeScript/React 规范

  • 使用 ESLint + Prettier
  • 组件名 PascalCase
  • 变量/函数名 camelCase
  • 常量 UPPER_SNAKE_CASE
  • 必须添加类型定义
// ✅ Good
interface UserProps {
  name: string;
  age: number;
}

const UserCard: React.FC<UserProps> = ({ name, age }) => {
  return <div>{name}: {age}</div>;
};

// ❌ Bad
function User(props) {
  return <div>{props.name}</div>;
}

2.3 Python 规范

  • 遵循 PEP8
  • 使用 Black 格式化
  • 类型注解必须
  • Docstring 格式：Google Style
# ✅ Good
def calculate_score(user_id: int) -> float:
    """Calculate user score based on activities.

    Args:
        user_id: The user ID.

    Returns:
        The calculated score.

    Raises:
        ValueError: If user not found.
    """
    user = get_user(user_id)
    if not user:
        raise ValueError(f"User {user_id} not found")
    return user.score

三、文档规范

3.1 代码注释

  • 公共函数/接口必须有注释
  • 复杂逻辑必须说明 "为什么"
  • TODO 必须标注人和时间
// ✅ Good
// ProcessPayment handles payment logic including:
// 1. Validate order status
// 2. Call payment gateway
// 3. Update order status
// 4. Send notification
// TODO(chengfeng): Add retry logic for step 2 (2026-06-21)
func ProcessPayment(ctx context.Context, orderID int64) error {
    // ...
}

3.2 API 文档

  • 使用 Swagger/OpenAPI
  • 每个接口必须包含：描述、参数、响应、错误码
  • 示例请求/响应必须真实可用

四、测试规范

4.1 测试层级

类型 范围 覆盖率要求 运行时机
单元测试 函数/方法 ≥ 70% 每次提交
集成测试 模块间交互 ≥ 50% PR 时
E2E 测试 完整流程 核心流程 发布前

4.2 测试命名

// 格式：Test<FunctionName>_<Scenario>
func TestCalculateScore_ValidUser(t *testing.T) { }
func TestCalculateScore_UserNotFound(t *testing.T) { }
func TestCalculateScore_NegativeScore(t *testing.T) { }

五、安全规范

  • [ ] 不硬编码密钥（使用环境变量/配置中心）
  • [ ] SQL 必须参数化（防注入）
  • [ ] 用户输入必须校验
  • [ ] 敏感操作必须鉴权
  • [ ] 日志不打印敏感信息

违反规范的处理：首次提醒 → 二次警告 → 影响绩效
